# 使用和配置 Skill

## Skill 是什么

Skill 是一份**清晰、可执行的指令文档**，用来告诉 AI Agent——在什么条件下（When）、按照哪些步骤（How）、产出什么结果（What）。

简单说：**Skill = 给 AI 写的操作手册**。写好一个 Skill，AI 就能在特定场景下稳定地帮你完成任务，而不是每次都要重新描述需求。

> 举个例子：如果你每周都要让 Agent 帮你写周报，与其每次都重复说"要包含哪些模块、用什么格式"，不如写一个 Skill，以后一句"写周报"就搞定，输出格式和质量每次都一样。

## 如何打开 Skills 管理

Skills 入口在**每个项目内部**，每个项目的 Skills 都是单独配置的。进入项目后，点击「Skill Market」按钮打开 Skills 管理面板。

面板顶部有两个 tab：

| Tab | 说明 |
|-----|------|
| **Public** | 公共资源库，可以浏览和搜索社区共享的 Skills（支持关键词搜索和分类筛选） |
| **Private** | 你自己创建的 Skills |

## 从 Skill Market 添加 Skill

1. 打开「Skill Market」→ 切换到 **Public** tab
2. 通过分类标签（如 Tools、Development、Business、Data & AI 等）或搜索关键词找到想要的 Skill
3. 点击 Skill 卡片上的「**Add to Project**」按钮
4. Skill 即被添加到当前项目中，可直接使用

> 添加之后不需要做任何配置。下次你的对话触发了这个 Skill 的场景，Agent 会自动按照 Skill 里定义的流程来执行。

## 自己创建 Skill

你可以在和 Agent 对话中直接新建 Skill：

1. 在对话中让 Agent 帮你创建 Skill，新建的 Skill 会出现在 **Private** tab 中
2. 创建后可在需要的项目中添加使用
3. 也可以发布到 **Public**，让其他人搜索到并使用你的 Skill

## 怎么写好一个 Skill

一个好的 Skill 需要包含三个核心要素：

| 要素 | 说明 | 举例 |
|------|------|------|
| **When（触发条件）** | 什么情况下触发这个 Skill | "当用户要求生成周报时" |
| **How（执行步骤）** | 具体按什么步骤执行 | "1. 收集本周变更 2. 按模块分类 3. 生成摘要" |
| **What（输出结果）** | 最终产出什么 | "输出一份 Markdown 格式的周报" |

### 写 Skill 的核心原则

- **一个 Skill 只做一件事**——职责单一，不要把多个任务塞进一个 Skill
- **步骤要具体可执行**——"分析代码"太模糊，"检查函数是否超过 50 行"才是好指令
- **预设失败策略**——告诉 AI 遇到异常怎么办，比如"如果找不到文件，提示用户确认路径"
- **避免模糊用词**——用"必须""禁止"代替"尽量""可以考虑"

### 用 AI 帮你写 Skill（推荐）

最高效的方式是**让 AI 帮你写**：

1. **先让 AI 执行一次真实任务**——直接描述你的目标，让 AI 尝试完成
2. **让 AI 复盘**——任务完成后，让 AI 总结成功步骤、失败点、可固化的流程
3. **基于复盘生成 Skill**——让 AI 按 When / How / What 的结构生成 Skill
4. **你快速审核**——只需关注边界是否合理、步骤是否可执行，确认后保存即可

> 不需要一开始就写出完美的 Skill。先让 Agent 做一遍，做得好就让它把过程固化下来，做得不好就调整再来——几轮迭代下来，Skill 的质量会越来越稳定。

### 常见错误（避坑）

| ❌ 别这样 | ✅ 应该这样 |
|----------|-----------|
| 一个 Skill 塞太多功能 | 拆成多个独立的 Skill |
| "尽量使用 TypeScript" | "必须使用 TypeScript，禁止使用 JavaScript" |
| 不写失败处理 | 明确写"如果 XX 失败，则执行 YY" |
| 步骤含糊不清 | 每一步都具体、可验证 |

## 完整流程总结

```
添加使用：Skill Market → Public → Add to Project → 当前项目可用

自己创建：对话中创建 → 出现在 Private → 添加到项目使用 → 可发布到 Public
```
