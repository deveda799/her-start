管理员入门

你是团队负责人或组织创建者？从注册到建好组织结构，全流程在这里。

准备工作：下载导入模板

在注册之前，请先下载以下两个 Excel 模板，提前准备好需要导入的组织架构和成员名单：

🏢

部门结构模板 (Department Template)

填写部门层级关系和部门负责人，用 "/" 分隔层级，如 "CodeBanana/R&D"

📥 点击下载 dept-template.xlsx 👥

成员名单模板 (Member Template)

填写成员姓名、邮箱、所属部门等信息，邮箱为必填项

📥 点击下载 member-template.xlsx
ℹ️
请在注册流程的第 4 步之前填好这两个文件。上传后系统会自动发送邀请邮件给所有成员。

注册和初始引导

访问 www.codebanana.com，支持 Google / Apple / 邮箱注册。注册完成后，系统会进入引导流程。

第 1 步：选择用途类型

首次登录后，系统询问你打算如何使用 CodeBanana：

  • Personal — 个人使用
  • Team / Enterprise — 团队协作

作为管理员，请选择 Team / Enterprise。

Step 1：选择 Team / Enterprise

第 2 步：创建或加入组织

选择团队后进入组织设置：

  • Create New Organization — 创建新组织（管理员选这个）
  • Join Existing Organization — 加入已有组织

Step 2：选择 Create New Organization

第 3 步：填写组织信息

填写组织基本信息：

  • Organization Name（组织名称）：默认根据注册邮箱生成，Owner 可以修改
  • Role（角色）：Admin（Owner）
  • Team Size（团队规模）：选择人数范围

填写完成后点击 Create & Continue。

Step 3：填写组织名称、角色和团队规模

第 4 步：导入组织架构

点击 Import Organization，上传你在准备工作中填好的两个模板文件：

  1. 拖拽上传 部门结构模板（dept-template.xlsx）
  2. 拖拽上传 成员名单模板（member-template.xlsx）
  3. 上传完成后，系统会自动发送邀请邮件给所有成员，成员通过链接注册即可加入

Step 4：上传部门结构和成员列表 Excel

第 5 步：预览确认

导入前可以通过 Structure Preview 确认：

  • 部门层级是否正确
  • 成员分配是否准确
  • 确认无误后点击 Upload & Import 完成导入

Step 5：预览部门和成员列表后确认导入

组织角色

角色权限
Owner创建者，拥有所有权限：管理成员、控制 Agent 权限、管理计费
Member普通成员，可使用 AI 资源和参与协作

注意：每个用户只能拥有一个 Own Organization。每个项目必须属于一个组织，不同组织之间的项目完全隔离。

邀请成员

除了引导流程中的批量导入，还有两种日常邀请方式：

方式一：添加组织成员

在项目中直接添加已加入组织的成员，无需发送邮件邀请。

方式二：邀请外部成员

通过 Invite external contacts 功能邮件邀请外部协作者：

  • 外部成员直接进入项目群聊，不会加入组织
  • 权限仅限于被邀请的项目
  • 适合临时协作者、客户、外部合作伙伴

新用户入门

被邀请加入团队？按照这个流程 5 分钟就能开始协作。

访问 www.codebanana.com，支持 Google / Apple / 邮箱注册。

个人用户引导

如果你是个人使用（不加入任何团队），注册后只需要两步就能开始：

第 1 步：选择用途类型

选择 Personal（个人使用）。之后随时可以加入团队。

选择 Personal — 个人使用

第 2 步：选择你的角色

选择最符合你用途的角色，系统会据此推荐对应的项目模板分类，创建项目时更方便：

  • Research & Analysis — 行业研究、竞品分析
  • Documents — 报告、文档、写作
  • Websites & Apps — 网站和应用开发
  • Data & Dashboards — 数据分析、看板

选择最符合你的使用场景

选好后点击 Get Started，即可进入工作台开始使用。

加入组织

如果你需要加入团队组织协作，注册后的流程如下：

第 1 步：选择 Team / Enterprise

在第一步中选择 Team / Enterprise，然后选择 Join Existing Organization。

选择 Join Existing Organization

第 2 步：搜索目标组织并发送申请

  • 在搜索框中输入组织名称
  • 从搜索结果中找到目标组织，点击 Select
  • 点击 Send Application 发送加入申请

搜索组织名称 → 选择 → 发送申请

第 3 步：等待审批

申请提交后：

  • 系统向组织管理员发送审批通知
  • 管理员批准后，你会自动成为该组织的 Member
  • 工作空间会自动切换到该组织下
💡

申请发出后耐心等一下就好。审批通过会有通知提醒，不用反复刷新。如果着急，可以直接联系管理员让他在 Notification 里点通过。

ℹ️
如果是通过 Invite external contacts 邮件邀请进入的，你会直接加入项目群聊，无需审批，也不会加入组织。适合临时协作场景。

注意：同一时刻只能处于一个组织下。切换组织 = 切换整个工作空间（项目、文件、通讯录全部随之切换）。

4

参与协作

在 Discussion 中发表意见，用 Comment 批注代码，通过 Forward 传递信息。

创建你的第一个项目

从新建到和 Agent 对话，3 步开始你的第一个项目。

创建项目（3 步）

第 1 步：命名项目

  • 登录 https://codebanana.com/（支持谷歌、苹果、邮箱登录）
  • 在 Studio 页面点击+按钮，选择创建方式：
  • Create Project：从头新建
  • Import GitHub Repository：从 GitHub 仓库导入
  • 选择 Create Project 后，填写项目名称（必填）

Studio 页面：点击 + 按钮选择 Create Project 或 Import GitHub Repository

第 2 步：选择模板

项目命名后，进入模板选择页面：

  • 从头开始：选择Custom（默认选项），空白项目自由发挥
  • 按类目选择：左侧提供分类导航，按行业或需求筛选模板
  • All（全部）
  • Research & Analysis（研究分析）
  • Documents（文档）
  • Websites & Apps（网站与应用）
  • Data & Dashboards（数据看板）
  • 搜索模板：找不到合适的？直接在搜索框输入你想做的事情，比如"周报"、"官网"
💡

不确定选哪个也没关系，选Custom就好。项目创建之后随时可以调整方向，不用在这一步纠结太久。

第 3 步：点击 Create

模板选好后，点击右下角Create按钮，项目即创建成功。

填写项目名称、选择模板后点击 Create

现在你面前就是一个全新的项目空间——对话框、文件区、预览区都准备好了。在对话框里直接告诉 Agent 你想做什么就行，不需要任何特殊格式，像跟同事说话一样自然。

项目创建完成：左侧文件树 + 右侧 Agent 对话区

项目创建后

每个项目自动对应一个群聊，包含三种模式：

模式说明
Team Agent和 Agent 协作，让它帮你写代码、做项目。一次一人使用，群主控制权限
Private Agent你的私人 QA 助手——有问题直接问，它基于项目代码回答。无需授权，不影响团队
Discussion团队成员之间的讨论区，自由沟通，不涉及 Agent
💡

第一次用？建议先在 Team Agent 里打个招呼，比如"帮我做一个公司介绍页面"或者"帮我分析一下这份 Excel"，看看 Agent 怎么回应。很多用户就是从这一句话开始，慢慢发现原来 AI 能帮的比想象中多得多。

自动生成的 User Guide

项目创建后，文件树中会自动生成一个 User Guide 目录，里面包含 CodeBanana 核心功能的详细说明文档（中英文版本）。

这个目录有两个用途：

  • 给你看：随时翻阅，了解产品功能
  • 给 Agent 看：Agent 执行任务时会自动读取这些文档作为知识源，帮它更好地理解项目
📘

不用刻意去读 User Guide——它更大的价值是让 Agent 自带产品知识。当你问"怎么部署"时，Agent 已经知道该怎么做了。

邀请团队

项目群聊可通过右边栏人+图标，在 Invite tab 中添加组织内成员或邀请外部好友加入。一个人用也完全没问题，但如果有同事一起参与，协作的感觉会完全不一样。

在 Invite 面板中选择组织成员或添加外部成员

修改项目名称

进入 Studio → 项目卡片右键或点击···菜单 → 选择Rename

右键项目卡片，选择 Rename 修改项目名称

上下文共享

这是 CodeBanana 最强大的机制之一：上传到项目中的所有文件，自动成为整个团队和 AI 的共享知识。

  • AI 和团队成员都基于同一个 repo、同一批文件、同一份知识工作
  • 不需要反复贴文件、不需要手动同步
  • 新人加入项目后，所有历史资料都在，不用从零交接
💡

想象一下：产品经理上传了需求文档，设计师上传了设计稿，开发直接让 Agent 基于这些材料写代码——所有人看到的都是同一份真相，再也不会出现"你看的是哪个版本？"的问题。

信息转发（Forward 功能）

  • AI 会将多人讨论的上下文融入到后续 coding，减少重复沟通
  • 支持单独、批量转发到不同群、不同 agent、Discussion tab

设计到开发的全链路上下文共享（Preview 功能）

  • 设计师能理解正在开发什么
  • PM 能看到逻辑文件和变更
  • QA 能直接定位代码实现

配置 Skill

Skill = 给 AI 写的操作手册，一次创建多次复用。

Skill 是什么

Skill 是一份清晰、可执行的指令文档，用来告诉 AI Agent——在什么条件下（When）、按照哪些步骤（How）、产出什么结果（What）。

简单说：Skill = 给 AI 写的操作手册。写好一个 Skill，AI 就能在特定场景下稳定地帮你完成任务，而不是每次都要重新描述需求。

💡

举个例子：如果你每周都要让 Agent 帮你写周报，与其每次都重复说"要包含哪些模块、用什么格式"，不如写一个 Skill，以后一句"写周报"就搞定，输出格式和质量每次都一样。

怎么写好一个 Skill

一个好的 Skill 需要包含三个核心要素：

要素说明举例
When（触发条件）什么情况下触发这个 Skill"当用户要求生成周报时"
How（执行步骤）具体按什么步骤执行"1. 收集本周变更 2. 按模块分类 3. 生成摘要"
What（输出结果）最终产出什么"输出一份 Markdown 格式的周报"

写 Skill 的核心原则

  • 一个 Skill 只做一件事——职责单一，不要把多个任务塞进一个 Skill
  • 步骤要具体可执行——"分析代码"太模糊，"检查函数是否超过 50 行"才是好指令
  • 预设失败策略——告诉 AI 遇到异常怎么办，比如"如果找不到文件，提示用户确认路径"
  • 避免模糊用词——用"必须""禁止"代替"尽量""可以考虑"

用 AI 帮你写 Skill（推荐）

最高效的方式是让 AI 帮你写：

  • 先让 AI 执行一次真实任务——直接描述你的目标，让 AI 尝试完成
  • 让 AI 复盘——任务完成后，让 AI 总结成功步骤、失败点、可固化的流程
  • 基于复盘生成 Skill——让 AI 按 When / How / What 的结构生成 Skill
  • 你快速审核——只需关注边界是否合理、步骤是否可执行，确认后保存即可
💡

不需要一开始就写出完美的 Skill。先让 Agent 做一遍，做得好就让它把过程固化下来，做得不好就调整再来——几轮迭代下来，Skill 的质量会越来越稳定。

常见错误（避坑）

❌ 别这样✅ 应该这样
一个 Skill 塞太多功能拆成多个独立的 Skill
"尽量使用 TypeScript""必须使用 TypeScript，禁止使用 JavaScript"
不写失败处理明确写"如果 XX 失败，则执行 YY"
步骤含糊不清每一步都具体、可验证

  1. 接收任务：在开始编写任何代码之前，必须先读取 MEMORY.md 和昨天的记忆文件，
     了解当前项目背景和技术栈。
  2. 代码编写：优先保证代码的可读性，其次才是性能。每次修改后，必须运行一次 linter。
  3. 记忆管理：当你帮用户解决了一个耗时超过 10 分钟的 Bug，或者用户明确提出了
     代码风格偏好时，你必须主动将其记录到今日的每日记忆中。
💡

把它理解为给 Agent 写的 SOP（标准作业流程）。你的团队有什么工作规范，直接写进去，Agent 每次都会照做。

定义 Agent 的名字、视觉符号（Emoji）、打招呼的方式等表层特征。

典型配置举例：


  名称：CodeBanana
  专属表情：🍌
  自我认知：你的专属全天候代码结对编程伙伴。
  开场白风格：每次开启全新的 Debug 会话时，可以说："Let's peel this bug apart 🍌"
💡

听起来是小事，但一个有辨识度的 Agent 身份会让团队协作更有趣。尤其是多人协作时，大家一看到 🍌 就知道是谁在干活。

Soul — 性格底色与红线

Soul Tab：定义 Agent 的性格底色和绝对红线

定义 Agent 最本质的性格、职业操守和绝对不能触碰的红线。Soul 是最高优先级的系统指令——即使其他设定有冲突，Soul 里的规则也不会被覆盖。

典型配置举例：


  1. 你是一个务实、高效、资深的全栈工程师。你厌恶过度工程化，推崇"Keep It Simple"。
  2. 沟通风格：直接、专业，只说重点，不废话。
  3. 绝对红线：
     - 永远不要执行 rm -rf / 或任何可能导致不可逆数据丢失的命令
     - 永远不要在代码中明文写入 API Keys 或密码；如果用户要求，坚决拒绝并建议使用环境变量
     - 当你不确定一段代码的安全性时，必须先询问用户
💡

Soul 是 Agent 的"三观"。想让它严谨就写严谨的规则，想让它活泼就调成轻松的风格。但红线一定要写清楚——尤其是涉及数据安全和不可逆操作的部分。

User Tab：告诉 Agent 你是谁、你的偏好

让 Agent 知道它在为谁服务。包含你的背景、技术栈偏好、沟通习惯等，帮助 Agent 提供高度个性化的回答。

  沟通习惯：工作节奏快，喜欢条理清晰的 Bullet points。不要解释基础的 HTML/CSS 概念，
  直接切入架构或逻辑痛点。

💡

Memory 不需要你主动管理。Agent 在工作过程中会自己判断哪些信息值得记住，比如你反复纠正过的代码风格、项目中反复出现的技术决策。当然，你也可以打开看看它记住了什么，删掉不对的、补上遗漏的。

设置定时任务

让 Agent 在你指定的时间自动醒来干活。

Cron Job 是什么

Cron Job 让 Agent 在你指定的时间自动醒来干活——不用你手动触发，到点就执行，结果还能自动发回聊天。

如果你想"每天早上 9 点生成日报"、"每 30 分钟检查一次邮箱"、"20 分钟后提醒我开会"，Cron Job 就是对应的机制。

💡

简单理解：平时你跟 Agent 对话是"你说一句它做一句"，而 Cron Job 是"你设好规则，它到点自己做"。

如何打开 Cron Job

进入项目后，点击右上角的 ··· 按钮，在下拉菜单中选择 Cron Job。

第 1 步：点击 Cron Job 打开定时任务面板
第 2 步：进入 Cron Job 管理界面

打开后页面分为两个区域：

区域说明
Heartbeat（心跳）页面上半部分，按固定时间间隔批量执行一组任务
Scheduled Tasks（定时任务）页面下半部分，在精确时间点执行单个任务

Heartbeat 和 Scheduled Tasks 怎么选

两种方式适用于不同场景。不确定用哪个的时候，按这个思路判断：


  1. 这是一次性提醒吗？
     是 → 使用 Scheduled Tasks
     否 → 继续...
  
  2. 任务是否需要在精确时间运行？
     是 → 使用 Scheduled Tasks
     否 → 继续...
  
  3. 此任务能否与其他周期性检查批量处理？
     是 → 使用 Heartbeat（添加到 HEARTBEAT.md）
     否 → 使用 Scheduled Tasks
类型适合场景举例
Heartbeat多个小任务需要定期批量检查每 30 分钟检查一次邮箱 + 检查一次数据异常 + 检查一次服务状态
Scheduled Tasks精确时间执行、一次性提醒每天早上 7 点读邮箱、下周五下午 3 点提醒我开会
💡

区别很简单：Heartbeat 是"每隔一段时间把清单上的事全做一遍"，Scheduled Tasks 是"到了这个时间点做这一件事"。

Heartbeat — 心跳任务

如何配置

  • 在 Cron Job 页面上半部分，设置 间隔时间（比如 Run every 30 Minutes）
  • 点击 Apply 保存设置
  • 打开右侧的开关，Heartbeat 即开始运行
Heartbeat：设置间隔时间并开启开关

开启后，程序会在每个间隔时间到达时，自动执行 HEARTBEAT.md 文件里列出的全部任务。

编辑任务清单

HEARTBEAT.md 就是 Heartbeat 的任务清单。右上角有两个按钮：

按钮作用
Edit编辑任务清单，添加或修改要定期执行的任务
Reset恢复为系统默认状态
💡

把你需要定期检查的事情都写进 HEARTBEAT.md 就行。比如"检查邮箱有没有新邮件"、"检查数据看板有没有异常指标"——Agent 每隔一段时间就会把清单上的事全做一遍。

Scheduled Tasks — 定时任务

如何创建

  • 在 Cron Job 页面下半部分，点击 + New Task 按钮
  • 弹出创建面板，填写以下信息：
Scheduled Tasks：点击 + New Task 创建定时任务
字段说明
Task Name给任务起个名字（必填），比如"每日晨报"、"周五竞品监控"
Trigger Conditions任务触发的时机（点击+ Add Condition添加）
Execution Flow触发后具体执行什么任务（必填）
  • 填完后点击 Create Task 完成创建

触发条件（Trigger Conditions）

支持两种触发方式：

方式一：Scheduled（按时间触发）

选项含义举例
Daily每天的某个时间每天早上 7:00
Weekly每周几的某个时间每周五下午 5:00
Monthly每月某日的某个时间每月 1 号上午 9:00
Once一次性执行2026 年 4 月 5 日下午 3:00
Every固定间隔执行每隔 2 小时

方式二：File Watch（文件变化触发）

当指定文件夹下的文件发生变化时自动触发任务。可以选择监控的文件夹和具体事件类型（新增、修改、删除）。

💡

比如"当 /data 目录下有新的 Excel 文件上传时，自动分析并生成报告"——这就是 File Watch 的典型用法。

Execution Flow（执行内容）

在这里描述触发后 Agent 要做什么。用自然语言写就行，比如：

  • "读取邮箱中的未读邮件，按紧急程度排序，生成摘要发回聊天"
  • "搜索竞品 A、B、C 的最新动态，整理成对比表格"
  • "检查 /data 目录下新上传的文件，自动归类并更新索引"

完整流程总结


  Heartbeat：设置间隔时间 → Apply → 打开开关 → 编辑 HEARTBEAT.md 写入任务清单 → 到点自动执行全部任务
  
  Scheduled Tasks：New Task → 填写任务名 → 设置触发条件（时间 或 文件变化） → 填写执行内容 → Create Task → 到点自动执行

Request 和 Token

什么是 Request

一个 Request = 你发送给 Agent 的一次交互（一条 prompt）。不管是让它写代码、回答问题还是执行任务，都算一次 Request。每个套餐包含固定数量的月度 Requests。

什么是 Token

Token 是模型实际消耗的计算量，包括你的输入（prompt、上下文、文件）和 Agent 的输出（回复、生成的代码）。

  • 任务越复杂、上下文越长，消耗的 Token 越多
  • 使用更高级的模型也会消耗更多 Token
  • Token 用于计算超出套餐后的按用量计费

扣费机制

  • 扣费归属和 Agent 归属保持一致——项目的 Owner 可以管理 Agent 权限，扣的费用也从 Owner 的额度扣除

💡

简单说就是：谁创建的项目，谁买单。不管是你自己用 Team Agent 还是同事在你项目里用 Private Agent，都从你（Owner）的额度里扣。

Team Plan

Team Plan 面向需要团队协作的组织，由管理员（Owner）统一管理成员的权益和用量。

计费方式

  • 每位被添加到 Team Plan 的成员，每月享有 300 requests
  • 管理员可以为每位成员选择不同的虚拟机配置，账单根据虚拟机档位从 Owner 的信用卡扣除席位费
  • 套餐内 requests 用光后，会自动启用按用量计费模式，额度上限由管理员设置

添加成员到 Team Plan

差价计算方式：

🔥 团队协作最佳实践

掌握基础后，这些模式让团队真正发挥 AI 协作的威力。

实时透明协作

CodeBanana 解决「团队脱节与信息不可见」问题：

  • 可视化编码：实时看到同事和 Agent 写代码，就像 Figma 中看到光标移动——无需等待构建、截图或开会解释
  • 实时同步：多人一起写同一个 repo，没有冲突、分支、延迟引起的混乱；即写即显、提交状态同步、变更实时广播
  • 清晰的协作轨迹：所有讨论、修改、代码上下文都在一个空间内沉淀，再不会出现「没人知道最新状态」的情况

共享项目上下文

AI 和人类使用同一个完整上下文，解决 coding agent 太私人、团队难以共享的问题：

  • 全局上下文同步：AI、开发、产品基于同一个 repo、同一批文件、同一批知识，不需要手动贴文件
  • 记住团队讨论：AI 将多人讨论的上下文融入后续 coding，减少重复沟通
  • 全链路共享：设计师理解正在开发什么，PM 能看到逻辑文件和变更，QA 直接定位代码实现

推荐工作流（5 步法）

1

Discussion 对齐需求

PM / 设计 / 研发在 Discussion 中讨论方案、贴 PRD、对齐期望，达成共识。

2

把关键结论和需求描述转发到 Team Agent，Agent 基于讨论结果直接执行。

3

Private Agent 验证

不确定的技术方案先在 Private Agent 中验证可行性，确认后再切到 Team Agent 正式执行。

4

Comment 批注驱动评审

用 Comment 精确批注代码行，引用到 Chat，取代「你看看第 38 行」这种模糊反馈。

5

Git 节点管理

重要节点手动提交 Commit 并写清 Message，Agent 走偏时一键回滚代码 + Restore 对话。

❓ FAQ：账号与组织

注册登录、组织管理、套餐计费相关的常见问题。

注册与登录

支持哪些登录方式？

Google 一键登录、Apple 一键登录、邮箱注册 & 登录。忘记密码可通过邮箱找回。

国内能访问吗？

可以。国内网络可直接访问 www.codebanana.com；使用 VPN 速度更快。手机浏览器也可以访问。

一个邮箱可以注册多个账号吗？

不可以。一个邮箱只能绑定一个账号。

组织管理

一个人可以加入多个组织吗？

可以。你有且仅有一个 Own Organization（自己创建的），但可以加入多个 Work Organization。切换组织 = 切换当前工作空间。

组织之间的数据隔离吗？

完全隔离。组织 A 的项目对组织 B 的成员完全不可见。

如何邀请外部人员？

两种方式：① 邀请对方加入你的组织（需审批）；② 通过 Add external members 邀请到具体项目（直接加入群聊，不加入组织）。

Owner 和 Member 权限有什么区别？

权限OwnerMember
管理成员✅❌
控制 Agent 权限✅❌
管理计费✅❌
使用 AI 资源✅✅
参与项目协作✅✅

套餐与计费

免费套餐有什么限制？

Free 套餐每月 30 个 Requests，适合个人体验。超出后需升级或启用 On-demand 按用量计费。

Team Plan 怎么计费？

按人头收费（$30/人/月），每人每月 300 Requests。管理员在 Usage → Members 页面统一管理。费用从 Owner 信用卡扣除。

什么是 On-demand Usage？

套餐 Requests 用完后，可以在 Usage 页启用按用量计费。管理员可设置每人的用量上限。

Request Support 时 Token 消耗会增加吗？

是的。技术支持在群期间，Token 用量按 1.5 倍计算。

升级 / 降级套餐后怎么算？

升级立即生效，差价从信用卡扣除；降级余额暂存 Stripe 账户，用于抵扣后续账单。

❓ FAQ：Agent 相关

Agent 对话、配置、Skill 等功能的常见问题。

对话模式

Team Agent 和 Private Agent 有什么区别？

对比项Team AgentPrivate Agent
可见范围团队所有人仅自己
共享文件✅ 共享项目文件✅ 共享项目文件
操作权限每次仅一人操作独立操作，不受限
适合场景正式开发、团队交付个人探索、方案验证

Discussion 频道有 Agent 吗？

没有。Discussion 是纯人类讨论频道，适合需求对齐、方案评审、决策讨论。达成共识后可以 Forward 给 Team Agent 执行。

Forward 转发有什么规则？

  • 消息可以转发到当前群或其他群
  • 转发时需选择目标群的具体 Tab（Team Agent / Private Agent / Discussion）
  • Team Agent 仅支持转发到有 Agent 权限的群
  • 支持多选消息同时转发

Restore 回滚会丢失代码吗？

不会。Restore 只重置对话上下文到指定节点，代码文件不受影响。如果需要同时回滚代码，请配合 Git 的「Go back to this commit」使用。

Agent 配置

SOUL 和 IDENTITY 有什么区别？

SOUL 是 Agent 的性格底色和行为红线（最高优先级），IDENTITY 是名字、Emoji 等表层特征。SOUL 决定「怎么做事」，IDENTITY 决定「叫什么名字」。

MEMORY 需要我手动维护吗？

不需要。MEMORY 主要由 Agent 自己维护——当互动中产生重要信息时，Agent 会自动记录。你可以查看和纠正，但通常不需要手动编辑。

修改 Agent Config 后立即生效吗？

是的。修改 Basic Info 后，下一次 Agent 对话时即刻生效。不需要重启或重新加载。

Skill

Skill 和项目模板有什么区别？

项目模板是创建项目时的初始化配置（含代码和文件）；Skill 是可以随时添加到已有项目的能力包（含 SKILL.md 和相关文件）。Skill 更灵活，可叠加使用。

如何让 Agent 创建 Skill？

直接在 Chat 中告诉 Agent「把 XXX 做成 Skill」，Agent 会自动在 .skills 目录创建 SKILL.md 和相关文件。

发布到 Skill Market 后别人能看到我的代码吗？

发布到 Public Market 后，所有人可以看到 Skill 内容并安装使用。如果不想公开，保留为 Private Skill 即可。

❓ FAQ：部署与环境

Preview、Deploy、VM、终端相关的常见问题。

Preview 预览

Preview 链接别人能打开吗？

可以，但仅在 VM 运行时有效。VM 关闭（如长时间无操作自动休眠）后链接会失效。需要永久链接请使用 Deploy。

Preview 和 Deploy 有什么区别？

对比项PreviewDeploy
依赖 VM✅ 需要 VM 运行❌ 独立部署
链接有效期VM 运行时有效永久有效
适合场景开发调试交付分享
更新方式代码修改实时更新需重新 Deploy

如何查看当前所有可用端口？

点击左下角的 Service 图标，可以看到当前所有运行中的服务及对应端口。也可以在 Terminal 中使用 netstat 或 ss 命令查看。

Deploy 部署

Deploy 支持哪些类型的项目？

支持静态 HTML、Next.js、React、Vue 等前端项目。后端 API 服务目前依赖 VM 运行（使用 Preview）。

Deploy 后可以绑定自定义域名吗？

可以。Deploy 后通过 Vercel CLI 绑定自定义域名（如 xxx.codebanana.app），在 Terminal 中执行：

💻
vercel alias set <deployment-url> xxx.codebanana.app

Deploy 后修改了代码怎么办？

需要重新 Deploy。Deploy 是静态快照，不会自动更新。重新 Deploy 后链接不变，内容自动更新。

VM 虚拟机

每个项目的 VM 环境是什么？

每个项目分配独立的云端 VM，预装：

  • Node.js（npm / npx）
  • Python（pip）
  • Git
  • 常用 CLI 工具（curl、wget、jq 等）

如何安装自定义依赖？

通过内置 Terminal 直接安装：

  • npm install xxx（Node.js 依赖）
  • pip install xxx（Python 依赖）
  • apt install xxx（系统级工具，可能需要 sudo）

VM 会自动关闭吗？

长时间无操作后 VM 会自动休眠以节省资源。下次访问项目时自动唤醒。休眠不影响文件，但运行中的服务会停止。

VM 卡住了怎么办？

在 Service 面板中点击「重启 VM」按钮。如果仍有问题，可以使用 Request Support 获取技术支持。

Terminal 终端

Terminal 支持哪些命令？

几乎所有 Linux 命令都可用。常用的：npm / pip / git / curl / ls / cat / grep 等。

可以运行后台服务吗？

可以。使用 & 或 nohup 在后台运行服务。但注意 VM 休眠后后台服务会停止。
