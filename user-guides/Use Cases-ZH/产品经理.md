# 产品经理的 CodeBanana 使用指南

> 产品经理最头疼的事情之一，就是脑子里有了想法，但要经过"写 PRD → 排设计 → 等开发 → 改需求 → 再等开发"这个漫长的循环才能看到结果。在 CodeBanana 里，你可以跳过大部分中间环节，直接把想法变成可以点击的原型。

---

## 场景一：快速做竞品分析

### 什么时候会用到

- 老板突然问"竞品最近有什么动作"，你需要 30 分钟内给出一份像样的报告
- 产品立项前需要对比 3-5 个竞品，不想花一周去注册体验再写报告
- 开周会前需要更新竞品动态

### 怎么做

直接在 Team Agent 里说清楚你要对比什么：

> "帮我做一份协作工具的竞品分析，对比 Notion、飞书和语雀，重点看 AI 功能"

Agent 会自动搜索公开信息，整理出结构化的对比报告——产品定位、核心功能对比、定价策略、差异化打法、机会点和建议，全部帮你列好。

如果某个维度需要深挖：

> "再深入分析一下 Notion 的 AI 定价策略"

Agent 会搜索最新资料，补充到报告里。

不想每周手动查竞品？设一个定时任务：

> "每周一早上 9 点自动搜索这三个竞品的最新动态，整理成对比更新"

从此每周一开会前就有现成的竞品周报等着你，不用临时抱佛脚。

### 效果

以前做竞品分析：注册体验 + 查资料 + 写报告，至少一周。现在 30 分钟出完整对比报告，设好定时任务后，每周自动更新，你只需要花 5 分钟看结论。

## 场景二：把模糊的想法拆成可执行的需求

### 什么时候会用到

- 你有一个产品想法，但还停留在"大概要做什么"的阶段
- 需要给研发团队提供结构化的需求文档
- 评审会前需要快速整理出用户故事和验收标准

### 怎么做

想法还不成熟的时候，可以先在 **Private Agent** 里一个人理一理——不会打扰团队，也不用担心半成品的想法被别人看到。等思路清晰了，再到 Team Agent 里让 Agent 动手。

把你的想法用最自然的语言告诉 Agent：

> "我们要做一个会员积分系统，用户消费得积分，积分可以兑换优惠券"

Agent 会帮你拆解成完整的用户故事，每个 Story 包含：
- **角色**（作为一个 XX 用户）
- **期望**（我希望 XX）
- **验收标准**（具体的 checklist）
- **优先级**和**工期预估**

还会帮你规划 Sprint 节奏——哪些先做（核心闭环），哪些后做（运营能力）。

觉得某个 Story 太粗？继续说：

> "Story 3 再拆细一点，兑换流程比较复杂"

Agent 会把它拆成 3 个子任务，每个带独立验收标准。

### 效果

以前写 PRD + 拆需求要 2-3 天。现在 10 分钟出用户故事，带验收标准和排期建议，直接丢给研发开干。

## 场景三：跳过 PRD，直接出可交互原型

### 什么时候会用到

- 你有个想法想快速验证，不想写几十页 PRD 才能看到效果
- 老板说"下午给我看个 Demo"
- 需要做用户测试，要一个能点击的原型

### 怎么做

告诉 Agent 你要几个页面、大概长什么样：

> "帮我做个会员积分系统的原型，包括积分首页、兑换中心、兑换详情三个页面"

Agent 会生成可以点击的 HTML 页面——不是线框图，而是真正能交互的页面：
- 点击"去兑换中心"会跳转
- 点击"兑换"按钮会弹出确认
- 自动适配手机屏幕
- 可以直接转发链接给任何人体验

改需求也不用重新排期：

> "兑换中心加个搜索栏，积分首页加个积分即将过期的提醒"

1 分钟改好。满意了？直接点 **Deploy**，拿到一条永久链接——发给老板、发到群里、发给客户，谁都能打开体验。后面你再改原型，链接会自动更新，不用反复发新地址。

### 效果

传统路径：写 PRD → 排设计 → 出交互稿，至少 1-2 周。现在 30 分钟出可交互原型，老板和客户直接体验。需求变了？一句话就改好，不用重新走流程。

## 场景四：花 5 分钟调教 Agent，让它更懂产品经理

### 什么时候会用到

- Agent 总是给你写代码级的回答，你其实想要的是产品方案
- 每次都要重复说"用表格"、"别写代码"、"我是 PM 不是开发"
- 你希望 Agent 默认用产品经理的思维方式来回答问题

### 怎么做

进入项目，点右上角 **「···」** → **「Basic Info」**，调整这几个 tab：

**User（告诉 Agent 你是谁）：**

```
用户身份：产品经理，负责 B 端 SaaS 产品。
背景：熟悉原型设计和数据分析，不直接写代码。
沟通偏好：用表格和清单，别给我代码片段。输出需求用用户故事格式，带验收标准。
```

**Soul（设定 Agent 的工作风格）：**

```
你是一个产品思维驱动的 AI 助手，擅长把模糊的想法变成结构化方案。
- 输出需求时必须包含：用户角色、期望行为、验收标准
- 做对比分析时必须用表格
- 任何方案都要给出"推荐做法"和"不推荐做法"
- 不要给出代码实现，除非我明确要求
```

**Agents（定义工作流）：**

```
1. 收到需求后，先确认目标用户和核心场景，不确定就先问
2. 输出方案前先列大纲，确认后再展开
3. 竞品分析必须包含：产品定位、核心功能、定价、差异化建议
```

设好之后，Agent 的每一次回答都会自动带上这些"底色"——输出格式更符合 PM 习惯，不用每次重复提。

### 效果

从"每次都要跟 Agent 解释我是产品经理"，变成"它比我的实习生还懂我要什么格式"。5 分钟设置一次，后面所有对话都受益。

## 给产品经理的小建议

1. **Private Agent 先想，Team Agent 再做**：想法不成熟的时候在 Private Agent 里自己先理清楚，确认了再到 Team Agent 让 Agent 动手。这样不会浪费团队的注意力，产出质量也更高。

2. **不用一步到位**：先出一个粗糙的版本，看看感觉对不对，再逐步调整。Agent 不怕改——你改 10 次它也不会有情绪。

3. **原型做好就部署**：别截图发微信了。部署拿到链接，老板手机上直接打开体验，比任何截图都有说服力。而且链接是永久的，你改了原型它自动更新。

4. **重复的事交给定时任务**：竞品监控、行业资讯整理这种每周都要做的事，设成 Cron Job 里的定时任务让 Agent 自动跑。按时间触发选 Scheduled Tasks（比如"每周一 9 点"），按频率触发用 Heartbeat（比如"每 2 小时查一次行业热点"）。

5. **Basic Info 设一次，长期受益**：花 5 分钟在 Basic Info 里写清楚你是 PM、喜欢什么格式、不要什么内容。这比你每次对话都重复说一遍效率高十倍。
